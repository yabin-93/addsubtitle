# AddSubtitle 现有接口整理需求文档

## 1. 文档说明

本文档基于当前仓库中的接口 YAML、接口封装代码和自动化测试用例反向整理而成，目的是把“现有接口已经支持什么能力、依赖什么前置条件、对外表现出什么业务规则”整理成一份可读的中文需求说明。

文档依据：

- `api_yaml/**`
- `api_moudle/**`
- `test_case/**`
- `common/**` 中与登录、数据库、测试数据相关的公共能力

说明：

- 本文档描述的是“当前代码已体现的需求”，不是从零设计的新需求。
- 某些枚举值的业务名称在代码里没有明确说明时，文档会标注“推断”或“待补充”。
- 返回体字段仅整理当前代码和测试里实际依赖的部分，不代表后端完整响应模型。

## 2. 系统目标

当前系统围绕“上传视频并生成双语字幕项目”展开，核心业务链路如下：

1. 用户通过邮箱验证码登录。
2. 用户创建“加字幕”项目。
3. 用户上传视频、封面、音频素材，等待媒体处理完成。
4. 用户查看并编辑原文字幕、译文字幕。
5. 用户可调整字幕显示方式、项目比例、背景、每行字数和字幕样式。
6. 用户可在时间轴上控制视频可见性、拖拽字幕起始时间。
7. 用户可查询项目列表、空间素材列表，并发起项目导出。

## 3. 全局业务约束

### 3.1 鉴权要求

- 除登录相关接口外，其余接口均要求在请求头中携带 `Cookie: talecast_token=...`。
- 自动化封装中支持 Cookie 失效后自动刷新并重试一次，但从接口行为看，未登录或无效 Cookie 仍应返回 `401` 或业务失败。

### 3.2 通用成功判定

当前代码默认将以下条件视为成功：

- HTTP 状态码为 `200`
- 响应体中 `success == true`
- 多数接口还会返回 `msg == "success"`
- 部分写操作接口会返回 `code == 0`

### 3.3 通用失败判定

当前已验证的失败场景包括：

- 登录验证码错误：HTTP `500`，业务码 `-3`
- 项目 ID 无效：HTTP `500`，业务码 `-33`
- 无效 Cookie：通常返回 HTTP `401`
- 部分编辑接口依赖 `sessionId`，若会话失效，封装层会重新获取 `sessionId` 后重试

### 3.4 关键业务标识

- `projectId`：项目唯一标识
- `sessionId`：项目编辑态会话标识，字幕编辑、样式编辑、时间轴编辑、导出进度查询等场景会使用
- `subtitleArrId`：同一条字幕在原文轨和译文轨之间的关联标识

### 3.5 异步处理要求

系统中多个能力不是同步完成，需要轮询确认结果：

- 媒体上传后等待视频、雪碧图、封面可用
- 字幕识别和翻译结果生成
- 样式、背景、比例、每行字数等配置更新生效
- 导出任务完成

当前自动化中使用的典型超时时间：

- 媒体就绪：30 到 60 秒
- 字幕就绪：90 秒
- 翻译结果：120 秒
- 导出完成：900 秒

### 3.6 本地依赖

当前项目创建链路依赖以下本地能力：

- `ffprobe`：读取视频时长、轨道等媒体信息
- `ffmpeg`：从视频生成封面、提取音频
- 默认测试文件：
  - 视频：`test_data/upload_files/video/英文30S.mp4`
  - 音频：`test_data/upload_files/audio/audio.mp3`
  - 图片：`test_data/upload_files/image/cover.png`
- 字体样式和背景切换场景可选依赖数据库；若数据库未配置，相关测试会回退到默认数据或跳过

#### 4.3.3 项目创建链路需求

基于现有实现，创建“加字幕”项目的完整需求链路如下：

1. 先调用预创建接口拿到 `projectId`。
2. 上报 `project_create` 事件。
3. 调用正式创建接口，传入项目名、语种、翻译语种、模式等信息。
4. 若用户上传视频，则需要补发 `project_edit` 和 `loading_start` 事件进入编辑态。
5. 客户端需要将本地媒体按 5 MB 分片上传到预签名 URL。
6. 分片上传完成后，调用后端回调接口确认上传结束。
7. 轮询项目状态、项目详情、项目媒体源，直到视频、雪碧图、封面全部可用。

补充规则：

- 当前默认项目模式为 `add_subtitle`。
- 当前默认源语言为 `English`，翻译语言为 `Chinese`。
- 当前默认 `useTranslate=true`、`lipSync=false`、`generateSpeech=false`。
- 若视频本身包含音频流，客户端会自动从视频提取音频并上传。
- 当前自动化要求上传后项目详情中的封面地址应为 `.png`，媒体源中的视频地址应为 `.mp4`，雪碧图地址应为 `.jpg`。

#### 4.3.4 `status` 接口字段完成顺序

基于当前代码，`/api/project/{projectId}/status` 的字段完成顺序可以整理为下面这套“已验证顺序”：

1. `projectId` 已创建  
   说明：项目在 `preCreate` 和 `create` 成功后，`status` 接口即可基于该 `projectId` 查询。

2. `videoUpload = true`  
   说明：这是当前代码里最先被当成“视频已挂载到项目”的完成信号。  
   同时要求：`/source` 接口里已经能取到 `data.video.url`。

3. 进入“媒体处理完成”阶段  
   当前代码要求以下条件同时成立才算项目媒体真正可用：
   - `videoUpload = true`
   - `videoCompleted = true`
   - `sprite = true`
   - `/source` 中 `data.video.url` 存在
   - 根据 `vttUrl` 推导出的雪碧图地址存在
   - 如果要求封面，则 `/detail` 中 `thumbnail` 存在

4. 资源可访问校验通过  
   说明：不仅状态字段要完成，当前代码还会继续校验资源地址是否真的可访问：
   - 视频地址可访问，且 `Content-Type = video/mp4`
   - 雪碧图可访问，且 `Content-Type` 以 `image/jpeg` 开头
   - 如果要求封面，封面地址可访问，且 `Content-Type = image/png`

需要单独说明的点：

- 当前代码只能明确证明 `videoUpload` 先于“完整媒体可用”阶段。
- 对于 `videoCompleted` 和 `sprite`，当前代码只要求两者在最终阶段同时满足，**没有单独断言谁一定先完成**。
- `projectCreatedFailed` 不是“完成顺序”字段，而是“失败状态”字段。当前在切换比例、切换背景后会校验它始终为 `false`。

### 4.4 字幕管理与样式配置

#### 4.4.1 字幕内容管理

| 功能 | 接口 | 方法 | 关键入参 | 当前需求说明 |
| --- | --- | --- | --- | --- |
| 获取项目字幕 | `/api/project/{projectId}/subtitle` | GET | `projectId` | 返回 `oriList` 与 `transList`，即原文轨和译文轨 |
| 批量编辑字幕 | `/api/project/{projectId}/batchEditSubtitle` | POST | `sessionId`、`list` | 支持批量编辑原文或译文字幕文本，也可用于更新时间轴起始时间 |
| 新增字幕 | `/api/project/{projectId}/addNewSubtitle` | POST | `sessionId`、`subtitleArrId`、`startTime`、`speechId`、`preSubtitleArrId`、`oriSegments`、`transSegments` | 在指定字幕后插入一条新字幕，并同时创建原文轨与译文轨数据 |
| 触发批量翻译 | `/api/project/{projectId}/subtitleBatchTranslate` | POST | `sessionId`、`arrIds` | 对指定字幕触发翻译任务 |
| 按字幕组查询 | `/api/project/{projectId}/getSubtitleByArrIds` | POST | `arrIds` | 用于验证新增、翻译、删除结果 |
| 删除字幕 | `/api/project/{projectId}/delSubtitle` | POST | `sessionId`、`subtitleArrId` | 删除一组字幕，原文轨和译文轨都应被移除 |

补充规则：

- 字幕数据以 `subtitleArrId` 关联原文和译文。
- `batchEditSubtitle` 既可修改译文，也可修改原文；修改原文后可再触发翻译接口刷新译文。
- 当前测试要求项目字幕就绪后，`oriList` 和 `transList` 都必须非空。
- 新增字幕后，原文轨和译文轨都应可查询到该 `subtitleArrId`。
- 删除字幕后，原文轨和译文轨都不应再出现该 `subtitleArrId`。

#### 4.4.2 字幕显示与项目设置

| 功能 | 接口 | 方法 | 关键入参 | 当前需求说明 |
| --- | --- | --- | --- | --- |
| 切换字幕显示模式 | `/api/project/{projectId}/updateSubtitleShow` | POST | `subtitleShowEnum` | 支持切换字幕显示模式，详情接口中的 `subtitleShow` 应回写最新值 |
| 切换项目比例 | `/api/project/{projectId}/updateInfo` | POST | `sessionId`、`aspectType` | 支持切换项目比例，详情接口中的 `aspectType` 应回写最新值 |
| 切换项目背景 | `/api/project/{projectId}/background` | POST | `sessionId`、`backgroundId` | 支持切换背景，详情接口中的 `backgroundId` 应回写最新值 |
| 更新每行字数 | `/api/project/{projectId}/updateCharNum` | GET | `charNum` | 支持调整每行最大字符数，若详情接口返回 `charNum`，则应同步更新 |

补充规则：

- `subtitleShowEnum` 当前只验证了 `0`、`1`、`2`、`3` 四种值。
- 根据测试标题推断，这四个值对应“原文 / 译文 / 双语 / 关闭”，但仓库内未见更明确的枚举注释，建议后续由产品或后端补齐。
- `aspectType` 当前只验证取值集合 `{0,1,2,3,4}`，具体每个数字代表的画幅名称，当前代码未给出明确定义。
- `backgroundId` 要求大于 `0`。
- `charNum` 要求大于 `0`。
- 切换比例和背景后，项目状态中的 `projectCreatedFailed` 仍应为 `false`。

### 4.5 时间轴编辑

补充规则：

- `updateVideo` 当前至少依赖以下字段：`id`、`volume`、`startTime`、`offset`、`trackOffset`、`duration`、`materialDuration`、`hideOrMute`、`trackIndex`、`type`、`visible`。

- 当前自动化允许字幕起始时间比较存在 5 ms 容差。

### 4.7 项目导出

| 功能 | 接口 | 方法 | 关键入参 | 当前需求说明 |
| --- | --- | --- | --- | --- |
| 发起导出 | `/api/project/{projectId}/exportBackend` | POST | `subtitleJson`、`headers`、`resolution`、`lipSync`、`deviceType` | 基于项目字幕生成导出任务 |
| 记录前端导出会话点 | `/api/project/{projectId}/projectPoint` | POST | `sessionId` | 前端导出前先提交当前编辑会话，用于进入导出链路 |
| 上报前端导出埋点 | `/api/internal/event_conform` | POST | `eventType=export_click`、`message`、`projectId` | 前端导出前上报 `export_click` 事件 |
| 发起前端导出 | `/api/project/{projectId}/exportFront` | POST | `sessionId` | 按前端编辑页链路发起导出，不再提交 `subtitleJson` |
| 查询导出进度 | `/api/project/{projectId}/exportProgress` | GET | `version`、`sessionId` | 轮询导出进度，进度达到 `100` 视为完成 |

补充规则：

- 后端导出接口使用 `multipart/form-data` 传参。
- 前端导出链路当前按浏览器抓包可确认的顺序为：`status -> projectPoint -> event_conform(export_click) -> exportFront`。
- 前端导出接口路径是 `/api/project/{projectId}/exportFront`，不是 `/api/project/{projectId}/exportFrontend`。
- `projectPoint` 和 `exportFront` 使用 `application/json` 传参，核心参数为 `sessionId`。
- 当前前端导出请求会同时携带 `Cookie: talecast_token=...` 和 `Authorization: Bearer <token>`。
- `subtitleJson` 仅后端导出需要，由原文轨和译文轨按 `subtitleArrId` 配对后生成。
- 每条后端导出字幕至少需要：
  - `startTime`
  - `duration`
  - `subtitleArrId`
  - `speed`
  - `words`
  - `translateWords`
- 当前默认后端导出参数：
  - `resolution = 1`
  - `lipSync = false`
  - `deviceType = "desktop"`
  - `version = 1`
- `exportProgress` 当前只在后端导出用例中轮询，现有抓包没有证明前端导出也复用这条进度接口。

## 5. 非功能需求

### 5.1 性能

- 项目列表接口响应时间应小于 5 秒。
- 空间管理相关列表接口响应时间应小于 5 秒。

### 5.2 稳定性

- 上传分片失败时，客户端应支持有限次重试。
- Cookie 失效时，客户端可重新登录后重试一次。
- `sessionId` 失效时，至少字幕设置类接口需要支持重新获取 `sessionId` 后重试。

### 5.3 数据一致性

- 项目名称修改后，项目列表应能查询到最新名称。
- 字幕新增、删除、翻译、样式修改后，查询接口应能在轮询窗口内反映最新状态。
- 比例、背景、字幕显示模式等项目配置修改后，项目详情应回写最新值。

## 6. 当前已知待补充项

以下内容在当前仓库中没有被完整定义，建议后续补齐正式说明：

1. `subtitleShowEnum` 数值与中文显示模式的正式映射。
2. `aspectType` 各枚举值对应的画幅名称与尺寸定义。
3. `/api/asset/original-video` 的请求 `type` 与响应 `data.type` 的正式映射关系。
4. 各接口完整响应体 schema。
5. 空项目名、超长项目名、错误邮箱格式等被跳过场景是否需要纳入正式需求。

### 7.4 默认回归链路总结

默认 P0 回归链路可以概括为：

1. 登录拿到 Cookie。
2. 创建“加字幕”项目并上传默认媒体。
3. 等待字幕生成并验证字幕编辑、翻译、样式、项目设置。
4. 验证时间轴编辑能力。
5. 验证项目首页列表和项目改名。
6. 验证空间管理相关列表。
7. 最后验证视频导出。

这条顺序说明当前测试设计是“先造数，再消费数据”，其中：

- `test_add_subtitle_create.py` 是后续大多数项目类测试的数据入口。
- `test_add_subtitle_subtitle.py` 和 `test_add_subtitle_timeline.py` 优先复用最新可用项目。
- `test_add_subtitle_export.py` 放在最后执行，避免导出过程干扰前面大量编辑类操作。

## 8. 结论

基于当前代码，系统已经形成一条较完整的“登录 -> 创建加字幕项目 -> 上传媒体 -> 获取并编辑双语字幕 -> 调整样式和时间轴 -> 查询素材空间 -> 导出视频”的业务闭环。现阶段最适合把这份文档作为“现状说明”和“后续补齐正式接口文档的基础版本”使用。
